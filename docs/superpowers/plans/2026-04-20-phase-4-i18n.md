# Phase 4 — i18n Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Add multi-language UI with `i18next` + `react-i18next`. Ship two locales (`en`, `fr`) out of the box with the namespaces the template already contains (`common`, `auth`, `validation`, `nav`, `settings`, `users`, `notifications`). Gate locale switching on `FF_I18N` (seeded in Phase 3). A tenant-level `i18n.supported_languages` setting controls which locales are offered to end-users. The existing auth routes and tenant shell are retranslated — all hardcoded English text moves to JSON files.

**Architecture (what arrives in this phase):**

- `app/utils/i18n.ts` — initialises `i18next` with resources, fallback `en`, cookie-based language detection (`i18n_lang`), 1-year TTL.
- `app/locales/{en,fr}/{common,auth,validation,nav,settings,users,notifications}.json` — seven namespace files per locale.
- `app/utils/config/settings-registry.ts` gains `i18n.supported_languages` (comma-separated list, default `"en,fr"`).
- `$tenant/_layout.tsx` reads the tenant's supported languages + current cookie language, hydrates the resolved language into an `<I18nProvider>` wrapper.
- `app/routes/resources/language-switch.tsx` — resource route that sets the `i18n_lang` cookie and redirects back.
- `app/components/language-switcher.tsx` — small dropdown rendered in the tenant top-nav when `FF_I18N` is on and the tenant has ≥2 languages.
- Existing auth + tenant routes swap inline strings for `t("…")` calls against the right namespace.

**Tech stack touched:** `package.json` (`i18next`, `react-i18next`, `i18next-browser-languagedetector`), `app/utils/i18n.ts` (new), `app/locales/**` (new), `app/root.tsx` (wrap children in the i18n provider), `app/routes/$tenant/_layout.tsx` (resolve language + pass down), `app/routes/resources/language-switch.tsx` (new), `app/components/language-switcher.tsx` (new), every `/routes/_auth/**.tsx` route (translations), `app/utils/config/settings-registry.ts` (one new setting).

## Decisions locked in this phase

| Decision                     | Choice                                                                                                                                                                             | Rationale                                                                                                                      |
| ---------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| Default locale               | `en`                                                                                                                                                                               | Template English-first. `fallbackLng: "en"` handles missing keys in other locales.                                             |
| Ship locales                 | `en` + `fr`                                                                                                                                                                        | Two is enough to validate the pattern; `es` trivial to add later by copying `fr/` and translating.                             |
| Locale namespaces (template) | `common`, `auth`, `validation`, `nav`, `settings`, `users`, `notifications`                                                                                                        | Only namespaces the template actually uses today. Apps built on the template add their own.                                    |
| Language persistence         | Cookie `i18n_lang`, 1-year TTL                                                                                                                                                     | Matches facilities. Works without auth; survives logout.                                                                       |
| Language detection order     | cookie → `accept-language` header → default                                                                                                                                        | Explicit user choice wins; otherwise respect browser preference.                                                               |
| Supported-languages setting  | `i18n.supported_languages`, string type, comma-separated codes                                                                                                                     | Matches facilities convention; parsed on read in the tenant loader.                                                            |
| Feature-flag gate            | `FF_I18N` (seeded in Phase 3, off by default)                                                                                                                                      | Language switcher hidden when flag is off. Translation machinery still loads (safe default) — users just don't see the picker. |
| Missing key handling         | `parseMissingKeyHandler` returns the key itself                                                                                                                                    | Matches facilities (and react-i18next default). Surfaces missing keys visibly to devs without crashing.                        |
| Namespace registration       | **Centralised in `app/utils/i18n.ts`**: every new namespace must be imported and added to both the `resources` object and the `ns` array. Document this in the readme + CLAUDE.md. | Facilities' pattern. Prevents the silent-raw-key-render failure mode.                                                          |
| Direction (LTR/RTL)          | `<html dir="ltr">` only this phase; `getLanguageDir(lang)` helper returns the right value but nothing wires it to `<html>` yet.                                                    | RTL support ships with Arabic/Hebrew locales later.                                                                            |
| RR integration style         | Plain i18next with browser-only `initI18n()` called once from `app/entry.client.tsx` (or equivalent); SSR fallback renders in `en` and rehydrates on the client.                   | Avoids the complexity of `remix-i18next` for a simple cookie-driven switch. Plain `i18next` is enough for the template.        |

> When adding a new locale namespace (e.g., `app/locales/en/work-orders.json`), you MUST also register it in `app/utils/i18n.ts`: import the JSON files for both `en` and `fr`, add them to the `resources` object under both languages, and add the namespace name to the `ns` array.
